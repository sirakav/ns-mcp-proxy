# Security policy

## Reporting a security vulnerability

**Please report security vulnerabilities privately** so we can fix them before public disclosure.

### How to report

Use GitHub [private vulnerability reporting](https://docs.github.com/en/code-security/security-advisories/guidance-on-reporting-and-writing-information-about-vulnerabilities/privately-reporting-a-security-vulnerability):

1. Open the repository on GitHub.
2. Go to **Security** → **Report a vulnerability**.

If you do not see that option, private reporting may not be enabled for this repository yet—repository admins should turn on **Settings** → **Code security and analysis** → **Private vulnerability reporting**.

Do **not** file **undisclosed** exploitable vulnerabilities as public issues or pull requests; that can put users at risk.

### What to include

- Affected version, tag, or commit (if known)
- Steps to reproduce the issue
- Potential impact
- Suggested fix or mitigation (optional)

### Scope

Reports should concern **this repository** and **official releases published from it** by the maintainers. Forks, unofficial builds, and third-party repackaging are **out of scope** unless the maintainers explicitly support them.

### Process and timelines

Maintainers will triage reports privately, work on a fix, and publish a [security advisory](https://docs.github.com/en/code-security/security-advisories/about-github-security-advisories) when appropriate (coordinated disclosure).

We aim to **acknowledge** new reports within **5 business days**. Time to fix depends on severity and complexity; we do not guarantee a resolution date in advance.

## For repository administrators

Enable **Private vulnerability reporting** under **Settings** → **Code security and analysis** so contributors can use **Security** → **Report a vulnerability** as documented above.
